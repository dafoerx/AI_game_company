# 项目总览 / Project Brief

## 1. 当前背景

项目方向已从“AI 游戏公司经营模拟”切换为更易验证原型体验的题材：

> **餐饮门店经营模拟**

目标是在 **Web** 平台上，用 **3 角执行模型 + AI 协作** 的方式，尽快做出一个可验证乐趣的原型，并据此判断是否继续扩展立项。

## 2. 已冻结决策

### 产品决策
- **主题**：餐饮门店经营模拟
- **平台**：Web
- **商业目标**：原型验证
- **版本形态**：浏览器中可运行的管理型原型 Demo

### 团队决策
- **执行模型**：3 角执行模型
- **角色结构**：`product-system` / `prototype-build` / `prototype-qa`
- **AI 定位**：辅助拆解、生成初稿、加速实现、补充校验
- **人工定位**：做范围判断、体验验收、质量把关和关键拍板

### 工具链决策
- **研发工作台**：OpenCode
- **主力模型**：GPT-5.3-Codex

## 3. 题材选择依据

本次切换到“餐饮门店经营模拟”，基于以下判断：

1. **更容易在短时间内讲清楚体验**
   - 玩家天然理解“接单、备餐、出餐、收钱、处理差评、升级设备”这些行为。
   - 不需要过多世界观解释，就能进入经营决策。

2. **更适合 Web 原型低成本表达**
   - 菜单、库存、订单队列、顾客满意度、设备状态都适合用表格、卡片和状态条表达。
   - 对连续动画、复杂空间表现的依赖较低。

3. **更利于快速验证核心循环**
   - 从开店到接单再到复购/扩张的正反馈链路比较短，适合 15～30 分钟原型验证。

4. **更容易裁剪 MVP**
   - 可以先只做一家店、少量菜品、少量顾客类型，也能跑通最小经营闭环。

## 4. 产品假设

### 玩家幻想
玩家想体验的是：
- 开一家小餐饮门店
- 在有限现金、库存、人手和出餐速度之间做取舍
- 用菜单组合和设备升级不断提升效率
- 通过服务质量与顾客满意度扩大影响力
- 从一家小店逐步走向稳定经营甚至扩张

### 核心循环
建议首版围绕以下循环展开：

**进货备料 → 接待顾客 / 接单 → 组织制作与出餐 → 获得收入与评价 → 处理库存与人手压力 → 升级菜单 / 设备 / 服务能力 → 开启下一轮营业**

### 首版验证点
- 玩家是否能快速理解订单、库存、现金和满意度之间的关系
- 玩家是否会因为菜单取舍、人手分配和设备升级而产生策略感
- 玩家是否愿意继续经营下一轮营业日
- Web 交互是否足够顺手，能支持连续 15～30 分钟体验

## 5. 设计原则

1. **先验证经营闭环，再补复杂内容**
2. **先跑通单店模型，再谈多店扩张**
3. **先做最小可玩原型，再补长文档**
4. **所有关键结论仍落 Git，但减少过程文档冗余**
5. **范围控制优先级高于内容丰富度**
6. **优先采用适合浏览器表达的系统与反馈**

## 6. 当前非目标

现阶段不把以下内容当作必须：

- 多店并行管理
- 强剧情包装
- 高保真角色演出
- 复杂装修自由度
- 大规模联网功能
- 完整商业化链路

## 7. 当前最重要的问题

接下来优先补齐：

1. 首版营业日经营闭环
2. 菜单 / 订单 / 库存的最小模型
3. 主界面信息架构
4. 首版数值框架（营业额、成本、满意度、损耗）
5. 核心页面与交互骨架

## 8. 当前结论

仓库接下来的重点应转向：

- 把餐饮题材下的核心循环与系统拆清楚
- 用 Web 低成本完成原型表达
- 用 3 个项目级 Skill 固化当前协作方式
- 用 OpenCode + GPT-5.3-Codex 推进小步快跑式研发
