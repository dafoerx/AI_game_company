# 项目总览 / Project Brief

## 1. 当前背景

项目方向已经从泛模拟经营，收敛为一个更适合当前条件的题材：

> **AI 游戏公司经营模拟**

目标是在 **3～5 人真人核心团队 + AI 员工协作** 的模式下，用 **Web** 形式快速做出一个可验证乐趣的原型，并据此判断是否继续扩展立项。

## 2. 已冻结决策

### 产品决策
- **主题**：AI 游戏公司经营模拟
- **平台**：Web
- **商业目标**：原型验证
- **版本形态**：可在浏览器运行的管理型原型 Demo

### 团队决策
- **核心团队规模**：3～5 人
- **当前推荐配置**：4 人核心团队
- **AI 定位**：辅助生产、方案生成、内容草拟、重复劳动提效
- **真人定位**：方向判断、体验验收、质量控制、关键拍板

### 工具链决策
- **研发工作台**：OpenCode
- **主力模型**：GPT-5.3-Codex

## 3. 题材选择依据

本次选择“AI 游戏公司经营模拟”，主要基于以下判断：

1. **题材已有成熟参照**
   - 游戏开发/公司经营题材本身具备吸引力，玩家容易对“立项、研发、测试、发行、扩张”形成代入。
   - 办公室、团队、产品、营销、服务器等模块的经营链条已经被同类作品验证具备可玩性。

2. **更适合 Web 原型形态**
   - 经营行为大多可以通过面板、数值、卡片、状态变化和简单动效表达。
   - 对复杂大地图、角色演出、连续动画的依赖较低，原型成本可控。

3. **与项目现实背景一致**
   - 当前团队本身就在讨论“AI 员工如何替代部分岗位”，把这个命题做进游戏主题里，会更聚焦，也更有辨识度。

4. **验证效率更高**
   - 即便先用低保真美术，也能先验证核心循环、资源流、成长节奏和决策压力是否成立。

## 4. 产品假设

### 玩家幻想
玩家想体验的是：
- 从一个小团队起步
- 管理有限的真人骨干与 AI 员工
- 选择项目方向并分配研发资源
- 处理现金流、算力、口碑、效率和风险之间的平衡
- 最终把小团队做成有影响力的 AI 游戏公司

### 核心循环
建议首版围绕以下循环展开：

**招募/配置团队 → 选择项目 → 分配人力与 AI 资源 → 推进研发 → 获取收入与口碑 → 升级工具链/团队能力 → 解锁更高目标项目**

### 首版验证点
- 玩家是否能快速理解每项资源的作用
- 玩家是否会因为取舍而产生策略感
- 玩家是否有继续扩张与优化配置的动机
- Web 交互是否足够顺手，能支持连续 15～30 分钟体验

## 5. 设计原则

1. **先验证核心循环，再扩张内容**
2. **先保体验闭环，再补系统复杂度**
3. **AI 负责提效，不负责最终拍板**
4. **文档与过程记录持续归档到仓库中**
5. **范围控制优先级高于“功能丰富”**
6. **优先采用适合浏览器表达的系统与反馈**

## 6. 当前非目标

现阶段不把以下内容当作必须：

- 大规模联网功能
- 强剧情叙事
- 高成本角色演出
- 复杂 3D 场景
- 完整商业化功能链路

## 7. 当前最重要的问题

虽然主题、平台与商业目标已经明确，但后续仍需尽快补齐：

1. 首版资源系统定义
2. 主界面信息架构
3. 项目/订单/产品系统的最小模型
4. 首版数值框架
5. 核心页面与交互原型

## 8. 当前结论

仓库接下来的重点应转向：

- 把主题下的核心循环与系统拆清楚
- 用 Web 低成本完成原型表达
- 用项目级 Skill 固化各角色工作方式
- 用 OpenCode + GPT-5.3-Codex 推进小步快跑式研发
