# 研发工具链 / Dev Toolchain

## 1. 已冻结工具链

### 工作台
- **OpenCode**

### 主力模型
- **GPT-5.3-Codex**

这意味着当前研发工作流应优先适配：
- 小步任务拆分
- 明确输入/输出边界
- 文档先行
- 低耦合、可快速验证的原型开发方式

---

## 2. 工作方式约定

### 任务拆分原则
所有开发任务优先拆成可以一次性完成并验证的小任务：

- 一个页面
- 一个系统切片
- 一个状态流转
- 一个配置结构
- 一个测试目标

避免把需求写成含糊的大段愿景，而是写成：

- 目标是什么
- 不做什么
- 输入输出是什么
- 验收标准是什么

### 文档协作原则
任何关键决策都应至少落到以下两个位置之一：

- `docs/` 中的正式规划文档
- `docs/99-过程记录/` 中的阶段记录

### 角色协作原则
每个角色都优先通过项目级 Skill 进入工作状态，再输出对应产物，减少风格漂移和理解偏差。

## 3. Web 原型的默认工程建议

当前目标是 **原型验证**，因此建议优先采用轻量、前端单体式方案：

- **前端框架**：React
- **语言**：TypeScript
- **构建工具**：Vite
- **样式方案**：Tailwind CSS 或等价轻量方案
- **状态管理**：优先简单状态；复杂后再引入 Zustand 等轻量库
- **数据存储**：优先本地 Mock / JSON / localStorage；除非验证明确需要，否则暂不引入正式后端

> 这不是最终技术栈承诺，而是最利于快速验证的默认建议。

---

## 4. 给 GPT-5.3-Codex 的任务写法建议

建议所有研发提示词都遵循以下结构：

1. **任务目标**
2. **当前上下文**
3. **需要修改的文件或模块**
4. **验收标准**
5. **明确不做的内容**

### 示例
```text
目标：实现 Web 原型中的“项目指派面板”页面。
上下文：这是 AI 游戏公司经营模拟项目的 MVP，当前平台为 Web，目标是原型验证。
范围：只实现前端页面与本地状态，不接后端。
验收：能够显示项目列表、可分配员工、可更新进度、刷新后本地保留。
不做：权限系统、真实联网、复杂动画。
```

---

## 5. 当前最适合的研发节奏

### 周期建议
- 每周固定一次范围复核
- 每 2～3 天做一次可玩切片交付
- 每个切片必须可演示、可说明、可回退

### 输出优先级
1. 先做能玩的系统切片
2. 再做界面统一
3. 再做数值打磨
4. 最后补扩展内容

---

## 6. 当前不建议的做法

- 一开始就搭复杂后端
- 一开始就做大地图或重动画表现
- 同时推进过多系统分支
- 用过长、过泛的提示词驱动研发
- 在未验证循环前提前铺商业化功能
