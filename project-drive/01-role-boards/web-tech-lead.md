# web-tech-lead role board

**角色**：Web 技术负责人  
**当前阶段定位**：把系统设计转成可实现架构、可拆分任务和最短开发路径

## 开始前必读

1. `project-drive/00-active-context.md`
2. `project-drive/02-task-cards/in-progress/TASK-001_核心循环与资源流及首页信息架构.md`
3. `project-drive/04-meeting-notes/2026-03-26_TASK-001_系统设计评审请求.md`
4. `docs/04-研发工具链/DEV_TOOLCHAIN.md`

## 当前轮到你时要做什么

- 评审 `TASK-001` 的技术可行性
- 判断首版是否坚持前端单体 + 本地存储方案
- 给出最小实现架构：状态管理、时间推进、项目计算、事件系统
- 输出可直接开发的任务拆分方向

## 当前确认项

- 5 个核心资源是否适合单一前端状态树管理
- 时间推进是使用 tick、定时器还是手动推进更稳
- 项目 / 员工 / 公司三类数据结构是否足够支撑首版
- 拖拽分配、事件弹窗、本地存档是否在首版复杂度内

## 你需要给出的答复项

- 推荐的前端工程结构
- 状态管理与存档方案
- 首批开发切片建议（按页面 / 系统 / 状态流拆分）
- 需降级或延后的高风险点

## 推荐写回格式

```md
## Tech Lead 确认项
- 架构：...
- 状态管理：...
- 存档：...

## Tech Lead 答复项
- 可直接实现：...
- 建议降级：...
- 预计风险：...

## 下一角色
- visual-ui / producer
```

## 当前默认下一棒

- 正常评审链路：handoff 给 `visual-ui`
- 若发现设计复杂度过高：先回 `producer` 或 `systems-economy`
