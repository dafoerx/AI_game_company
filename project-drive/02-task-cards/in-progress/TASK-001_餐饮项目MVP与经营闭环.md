# TASK-001：餐饮项目 MVP 与经营闭环

**状态**: In Progress  
**负责人**: product-system  
**创建时间**: 2026-03-26  
**目标**: 用最小范围定义首版餐饮经营原型的闭环，为工程骨架提供直接输入。

---

## 1. 任务目标

把餐饮题材的首版原型压缩到可直接开工的程度，重点回答：

1. 玩家在一轮营业里做什么
2. 哪些资源必须可见
3. 哪些操作必须能点通
4. 哪些内容明确延后

## 2. 当前建议闭环

**营业前准备 → 接单 → 制作 / 出餐 → 获取收入与评价 → 处理库存 / 人手压力 → 升级菜单 / 设备 → 下一轮营业**

## 3. 首版必须明确的资源

- 现金
- 库存
- 员工人手 / 精力
- 顾客满意度
- 当前营业时段

## 4. 首版必须可操作的动作

- 开始营业
- 接受订单
- 分配工位或员工处理订单
- 完成出餐并结算
- 补货 / 调整库存
- 结束营业并查看结算

## 5. 首版必须可见的信息

- 当前现金
- 订单队列
- 库存余量
- 工位 / 员工状态
- 满意度 / 差评提示
- 当前营业进度

## 6. 明确延后项

- 多店经营
- 深度装修
- 复杂剧情与角色支线
- 复杂活动营销系统
- 复杂供应链模拟
- 复杂自动化测试体系

## 7. 当前待确认

- 首版是优先做堂食、外卖，还是只保留一种订单类型
- 菜品数量是否先压到 3～5 个
- 人手分配是按“员工卡片”还是按“工位按钮”实现
- 营业时间推进按实时、回合，还是半回合推进实现

## 8. 验收标准

完成后应满足：
- `prototype-build` 无需再追问主闭环是什么
- 可以直接开始搭页面骨架和状态结构
- 延后项明确写出，不再反复讨论
- 所有字段至少达到 mock 数据可实现粒度

## 9. 下一位角色

- 默认下一位：`prototype-build`
- 触发条件：本任务补齐首版实现边界并冻结
