## 1. 任务目标

把餐饮题材的首版原型压缩到可直接开工的程度，重点回答：

1. 玩家在一轮营业里做什么
2. 哪些资源必须可见
3. 哪些操作必须能点通
4. 哪些内容明确延后

## 2. 当前建议闭环

**营业前准备 → 接单 → 制作 / 出餐 → 获取收入与评价 → 处理库存 / 人手压力 → 升级菜单 / 设备 → 下一轮营业**

## 3. 首版必须明确的资源

- 现金
- 库存
- 员工人手 / 精力
- 顾客满意度
- 当前营业时段

## 4. 首版必须可操作的动作

- 开始营业
- 接受订单
- 分配工位或员工处理订单
- 完成出餐并结算
- 补货 / 调整库存
- 结束营业并查看结算

## 5. 首版必须可见的信息

- 当前现金
- 订单队列
- 库存余量
- 工位 / 员工状态
- 满意度 / 差评提示
- 当前营业进度

## 6. 明确延后项

- 多店经营
- 深度装修
- 复杂剧情与角色支线
- 复杂活动营销系统
- 复杂供应链模拟
- 复杂自动化测试体系

## 7. 当前待确认

- 菜品数量是否先压到 3～5 个

- 营业时间推进按实时、回合，还是半回合推进实现

## 8. 验收标准

### 风险缓解建议

#### 复杂度控制机制
1. **菜品系统上限**：
   - 首版锁定3个菜品（汉堡、薯条、饮料）
   - 每个菜品有固定配方（2-3种食材）
   - 解锁新菜品需要达到特定满意度或现金目标

2. **员工系统简化**：
   - 固定2个工位（1厨师+1服务员）
   - 员工无等级成长，只有疲劳状态
   - 休息机制：空闲时自动恢复疲劳

3. **库存管理简化**：
   - 3种基础食材（面包、蔬菜、肉）
   - 统一进货价格（$10/份）
   - 自动补货提醒，手动确认购买

4. **时间推进简化**：
   - 点击"下一订单"推进游戏
   - 每个订单固定耗时（厨师制作时间+服务员出餐时间）
   - 每日固定10个订单后自动进入结算

#### 首版验证边界
**必须验证**：
1. 玩家是否能理解"接单→制作→出餐"循环
2. 玩家是否会主动管理库存（避免售罄）
3. 玩家是否能感受到员工分配的重要性
4. 玩家是否愿意进行下一轮营业

**暂不验证**：
1. 多店扩张乐趣
2. 复杂装修系统
3. 深度员工成长
4. 供应链优化策略
5. 营销活动效果

### QA补充建议（仅供参考）

#### 建议的营业日阶段划分
```
准备阶段 → 营业中（循环） → 结算阶段
```
**详细状态**：
1. **准备阶段**：
   - 检查库存余量
   - 补充库存（消耗现金）
   - 分配员工到工位
   - 确认今日菜单（开启/关闭某些菜品）

2. **营业中（主循环）**：
   - 顾客到达，生成订单
   - 分配订单到空闲工位/员工
   - 制作菜品（消耗库存+时间）
   - 出餐（获得现金+满意度影响）
   - 处理异常（库存不足、员工疲劳、顾客等待超时）

3. **结算阶段**：
   - 计算当日总收入
   - 计算成本（库存消耗、工资）
   - 更新顾客满意度累计
   - 显示经营报告

#### 建议的核心资源定义
**现金**：
- 类型：货币，支持小数
- 来源：订单收入
- 消耗：进货成本、员工工资、设备升级
- 预警：现金<100时警告

**库存**：
- 类型：食材库存（如面粉、蔬菜、肉类），非菜品库存
- 来源：进货购买
- 消耗：制作菜品时按配方消耗
- 预警：任意食材<3份时警告

**员工**：
- 类型：厨师（制作）、服务员（接待/出餐）
- 状态：空闲、忙碌、疲劳
- 效率：制作速度系数（1.0为基准）
- 疲劳：连续工作后效率下降，休息恢复

**满意度**：
- 范围：0-100（累计值）
- 正面影响：快速出餐、菜品质量高
- 负面影响：等待超时、菜品售罄
- 影响：口碑效应，影响顾客到达率

#### 建议回答待确认问题
1. **订单类型**：首版只做堂食（简化配送逻辑）
2. **菜品数量**：3个基础菜品（如汉堡、薯条、饮料）
3. **分配方式**：按工位按钮（点击工位分配订单）
4. **时间推进**：半回合制（点击"下一订单"推进）
