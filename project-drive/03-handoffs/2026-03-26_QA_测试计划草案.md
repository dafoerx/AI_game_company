# TASK-001 QA测试计划草案

**创建者**：ai-game-company-prototype-qa  
**日期**：2026-03-26  
**状态**：草案，待评审  
**关联任务**：TASK-001核心循环与资源流设计

## 一、测试目标

基于TASK-001设计，验证以下核心假设：
1. 玩家能在5分钟内理解游戏基本操作和资源含义
2. 玩家会在15分钟内感受到有意义的策略取舍
3. 玩家在30分钟体验后仍有继续游戏的意愿
4. 失败条件合理，既不过于严苛也不过于宽松

## 二、核心验证指标

### 1. 前5分钟理解度指标
| 指标 | 测量方法 | 目标值 | 备注 |
|------|----------|--------|------|
| 操作成功率 | 记录首次完成项目分配所需的操作步骤数 | ≤3步 | 理想：点击项目→点击员工→确认分配 |
| 资源识别率 | 测试后问卷：正确说出5种资源用途的比例 | ≥80% | 现金=生存，算力=产能，士气=效率，口碑=溢价，进度=产出 |
| 目标理解度 | 测试后问卷：清晰说出游戏目标的比例 | ≥90% | 目标：经营AI游戏公司，扩张团队，完成项目 |
| 界面导航效率 | 首次找到关键功能的时间（项目管理、员工管理） | ≤60秒 | 通过录屏分析操作路径 |

### 2. 10～15分钟策略感指标
| 指标 | 测量方法 | 目标值 | 备注 |
|------|----------|--------|------|
| 首次决策思考时间 | 首次面临"招聘vs升级"选择时的思考时长 | 30-60秒 | 过短=无策略，过长=困惑 |
| 资源取舍意识 | 测试中主动提及资源限制的玩家比例 | ≥70% | 如："现金不够了"、"算力不足" |
| 策略多样性 | 不同玩家选择不同扩张路径的比例 | ≥3种 | 招聘优先、升级优先、平衡发展 |
| 风险感知度 | 玩家主动关注士气/现金预警的比例 | ≥60% | 显示对失败风险的意识 |

### 3. 30分钟继续意愿指标
| 指标 | 测量方法 | 目标值 | 备注 |
|------|----------|--------|------|
| 主动继续率 | 完成30分钟体验后主动选择继续游戏的比例 | ≥70% | 直接观察"继续游戏"按钮点击 |
| 扩张感知度 | 明确感受到效率提升的玩家比例 | ≥80% | 问卷："是否感觉公司变强了？" |
| 下一步计划 | 有明确下一步目标的玩家比例 | ≥75% | 如："我要招更多AI"、"升级服务器" |
| 满意度评分 | 整体体验满意度（1-5分） | ≥4.0 | 测试后问卷 |

### 4. 失败条件合理性指标
| 指标 | 测量方法 | 目标值 | 备注 |
|------|----------|--------|------|
| 失败预警感知 | 玩家注意到失败预警的比例 | ≥90% | 现金流预警、士气警告等 |
| 挽救机会感知 | 玩家认为有足够机会避免失败的比例 | ≥80% | 问卷："失败是否感觉公平？" |
| 新手失败率 | 首次体验30分钟内触发失败的比例 | 10-20% | 既不太容易也不太难 |
| 失败原因理解 | 能正确说出失败原因的玩家比例 | ≥85% | 如："现金花完了"、"士气崩溃了" |

## 三、测试方法

### 1. 新玩家观察测试
- **样本量**：8-12名目标用户（有游戏经验但未接触本游戏）
- **测试时长**：每人30分钟核心体验 + 10分钟问卷访谈
- **测试环境**：受控环境，屏幕录制，操作日志
- **观察重点**：
  - 首次困惑点及时间
  - 自然操作路径
  - 决策思考过程
  - 情绪反应点

### 2. 平衡性压力测试
- **测试方法**：模拟极端策略，验证系统稳定性
- **测试场景**：
  - 只招聘不升级策略
  - 只加班不休息策略
  - 只接大项目策略
  - 最小化支出策略
- **验证目标**：确保无死循环或无限刷取漏洞

### 3. 节奏感验证测试
- **测试方法**：记录各阶段实际时长，对比设计目标
- **阶段划分**：
  - 阶段1：0-3分钟（首次决策）
  - 阶段2：3-10分钟（第一次重要选择）
  - 阶段3：10-20分钟（首次扩张成果）
  - 阶段4：20-30分钟（MVP验证点）
- **调整依据**：如阶段偏差>30%，需调整数值平衡

### 4. 失败条件验证测试
- **测试方法**：故意触发各种失败条件，验证预警和后果
- **失败场景**：
  - 现金流断裂（连续3天现金为负）
  - 士气崩溃（士气<20持续1周）
  - 算力枯竭（算力为0持续24小时）
  - 口碑崩坏（口碑<0）
- **验证重点**：预警及时性、挽救可能性、后果合理性

## 四、测试用例清单

### 用例1：首次体验流程测试
```markdown
**前置条件**：新玩家首次进入游戏
**测试步骤**：
1. 创建公司，选择初始项目
2. 配置初始团队（1真人+2AI）
3. 分配AI员工到研发任务
4. 完成第一个小项目
5. 面临第一次扩张选择
**预期结果**：
- 15分钟内完成以上步骤
- 玩家理解基本资源含义
- 玩家感受到扩张乐趣
```

### 用例2：资源压力测试
```markdown
**前置条件**：游戏进行到第20分钟
**测试步骤**：
1. 同时启动2个中型项目
2. 持续高强度工作（不休息）
3. 观察资源变化（现金、算力、士气）
4. 触发至少一个资源警告
**预期结果**：
- 资源警告在合理时间点触发
- 玩家有足够时间采取措施
- 失败不会突然发生
```

### 用例3：策略多样性测试
```markdown
**前置条件**：3名不同策略倾向的玩家
**测试步骤**：
1. 玩家A：优先招聘AI员工
2. 玩家B：优先升级工具和服务器
3. 玩家C：平衡发展策略
4. 比较30分钟后的公司状态
**预期结果**：
- 三种策略都有可行性和优劣
- 无绝对最优策略
- 不同策略带来不同体验
```

### 用例4：失败恢复测试
```markdown
**前置条件**：玩家触发士气崩溃警告
**测试步骤**：
1. 士气降至25，触发警告
2. 玩家采取措施（团建、休息）
3. 观察士气恢复速度
4. 验证恢复成本合理性
**预期结果**：
- 警告在士气<30时触发
- 恢复措施有明显效果
- 恢复成本与收益平衡
```

## 五、数据收集方案

### 1. 自动收集数据
- **操作日志**：记录所有按钮点击、页面切换、操作序列
- **时间戳数据**：记录各阶段开始/结束时间
- **资源变化**：记录现金、算力、士气、口碑的变化历史
- **事件触发**：记录所有随机事件和玩家选择

### 2. 手动观察记录
- **困惑点记录**：观察员记录玩家困惑表情/言语
- **决策思考记录**：记录玩家重要决策前的思考时长
- **情绪反应记录**：记录兴奋、沮丧、满意等情绪点
- **口头反馈记录**：记录玩家的即时评论和反馈

### 3. 测试后问卷
```markdown
## 核心理解问题
1. 请描述游戏的5种主要资源及其作用
2. 游戏的主要目标是什么？
3. 扩张公司的主要方式有哪些？

## 策略体验问题
4. 你面临的最困难决策是什么？
5. 你觉得哪种资源最紧张？为什么？
6. 你下一步打算如何发展公司？

## 整体反馈问题
7. 整体体验满意度（1-5分）
8. 是否愿意继续玩？为什么？
9. 最大的改进建议是什么？
```

### 4. 数据分析工具
- **操作路径可视化**：生成玩家操作热力图
- **资源变化曲线**：绘制各资源随时间变化图
- **阶段时长分布**：统计各阶段实际时长分布
- **策略分类分析**：基于操作序列聚类玩家策略类型

## 六、成功标准

### 必须通过标准（阻断性）
1. **安全通过率**：≥95%玩家能完成30分钟体验而不崩溃
2. **基础理解率**：≥80%玩家能正确描述游戏基本目标
3. **操作成功率**：≥90%玩家能独立完成核心操作

### 期望达到标准（体验性）
1. **继续意愿率**：≥70%玩家愿意继续游戏
2. **策略感知率**：≥60%玩家感受到有意义的取舍
3. **扩张乐趣率**：≥75%玩家感受到扩张带来的乐趣
4. **失败公平率**：≥80%玩家认为失败条件公平合理

### 优秀表现标准（理想）
1. **高满意度**：平均满意度≥4.2/5.0
2. **强策略深度**：出现≥4种明显不同的可行策略
3. **自然学习曲线**：90%玩家无需外部帮助完成学习
4. **情绪积极**：积极情绪点数量 > 消极情绪点数量

## 七、风险与缓解措施

### 高风险：玩家不理解资源系统
- **缓解措施**：
  1. 增加资源教学提示（前3分钟）
  2. 优化资源展示方式（图标+文字+颜色）
  3. 提供资源详情悬浮提示

### 中风险：失败条件过于严苛
- **缓解措施**：
  1. 增加多级预警（提醒→警告→危险）
  2. 提供明确的挽救指南
  3. 设置新手保护期（前15分钟）

### 低风险：策略单一化
- **缓解措施**：
  1. 平衡不同发展路径的收益
  2. 增加策略提示和对比
  3. 设计鼓励多样化的随机事件
