## Visual/UI 确认项

### 1. 界面架构确认
- **原设计**：三栏布局（顶部导航+左侧导航+中央工作区+底部操作栏）
- **技术建议**：单屏主界面+模态窗口（web-tech-lead建议）
- **评估**：✅ 支持单屏方案，更适合原型验证和快速理解

### 2. 信息表达清晰度
- **现金/算力/士气/口碑表达**：✅ 设计清晰，数值+进度条/百分比显示可行
- **项目与员工卡片**：✅ 卡片式展示适合，但字段需精简
- **警告与状态提示**：⚠️ 需要强化视觉反馈（颜色、图标、动画）

### 3. 交互复杂度评估
- **拖拽分配**：❌ 首版不建议（技术建议降级为点击分配）
- **多级导航**：❌ 首版不建议（技术建议单屏+模态窗口）
- **事件决策**：⚠️ 需要简化决策界面，避免选项过多

### 4. 信息密度评估
- **当前设计信息密度**：中等偏高（可能分散新手注意力）
- **MVP适宜密度**：低到中等（聚焦核心循环）
- **建议**：隐藏次要信息，按需展示

### 交互建议

#### 1. 项目分配流程（简化版）
**原设计**：拖拽员工到项目卡片
**建议方案**：
```
1. 点击员工卡片 → 显示"分配"按钮
2. 点击"分配" → 弹出项目选择列表
3. 选择项目 → 显示确认对话框
4. 确认 → 更新分配状态
```

**界面示例**：
```typescript
// 员工卡片组件
<EmployeeCard 
  employee={employee}
  onSelect={(emp) => setSelectedEmployee(emp)}
/>

// 项目选择模态窗口
<ProjectSelectModal
  projects={availableProjects}
  onSelect={(project) => assignEmployee(selectedEmployee, project)}
/>
```

#### 2. 时间推进交互
**原设计**：复杂的时间控制面板
**建议方案**：
- 主按钮："推进1周"
- 次级选项：下拉菜单选择时间跨度（1天、1周、1月）
- 视觉反馈：推进时显示资源变化摘要

#### 3. 事件处理界面
**原设计**：多选项复杂决策弹窗
**建议方案**：
- 简单事件：Toast通知（项目完成、资源变化）
- 重要决策：确认/取消对话框（投资机会、危机处理）
- 关键选择：最多3个选项的决策卡片

### 需要系统策划调整的内容

#### 1. 信息结构重组
**问题**：当前设计将太多信息放在首屏
**建议**：
- 将"公司状态概览"从主工作区移到侧边栏或折叠面板
- 将"待处理事件"从常驻区域改为通知中心
- 减少项目卡片上的非必要字段

#### 2. 交互流程简化
**问题**：部分交互流程过于复杂
**建议**：
- 统一采用"选择→操作→确认"三步流程
- 减少模态窗口的嵌套层级
- 合并相似功能的操作入口

#### 3. 视觉反馈强化
**问题**：状态变化反馈不足
**建议**：
- 资源变化时添加动画效果（数字滚动、进度条填充）
- 重要事件添加声音/视觉提示
- 错误操作提供明确的错误信息

#### 4. 移动端适配考虑
**问题**：当前设计主要针对桌面端
**建议**：
- 采用响应式布局，确保移动端基本可用
- 触控区域大小符合移动端标准
- 简化移动端操作流程

## 界面风险

### 1. 信息过载风险
**描述**：首屏信息过多，新手玩家难以聚焦
**缓解**：
- 采用渐进式信息展示（默认隐藏次要信息）
- 添加"简化视图"选项
- 强化视觉层次和分组

### 2. 交互混乱风险
**描述**：操作入口分散，玩家不知道下一步该做什么
**缓解**：
- 设计明确的主操作流程
- 添加上下文指引和工具提示
- 限制并行操作数量

### 3. 响应性能风险
**描述**：频繁的状态更新可能影响界面响应
**缓解**：
- 批量更新界面状态
- 使用虚拟滚动处理长列表
- 优化组件重渲染逻辑

### 4. 一致性风险
**描述**：不同界面风格和交互不一致
**缓解**：
- 建立设计系统（颜色、间距、组件）
- 创建交互模式库
- 统一错误处理和加载状态
