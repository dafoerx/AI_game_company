### 1) 本角色判断（visual-ui）

1. **TASK-003可进入UI落地，但当前应先“修订后启动”**：先统一活跃任务与验收口径（回合数、失败复现、日志字段），否则界面会反复改。  
2. **M2前半必须“可读性优先于特效完整度”**：赛博朋克风格保留，但扫描线、故障闪烁等高干扰效果默认弱化/可关闭，先保证信息层级清晰。  
3. **核心交互应收敛为单屏闭环**：建议固定为“建造/拆除 → 人口分配 → 结束回合（事件反馈）”，减少跳页与记忆负担。  
4. **即时反馈要做成“双层可见”**：  
   - 操作瞬间：资源与人口占用立即变化（数字跳变/颜色提示）；  
   - 下一回合：预测Δ资源与风险等级同步刷新。  
5. **失败路径必须有可视预警**：关键资源连续归零计数（如`1/2`）必须常驻可见，避免“无预警暴毙”，也方便QA复盘。

### 2) 对其他角色的冲突点/共识点

**共识点**
- 与 `producer`：同意“闭环优先、范围冻结、先可玩后精修”。  
- 与 `systems-economy`：同意把维护消耗、拆除返还、失败计数器显性展示，不做黑箱。  
- 与 `web-tech-lead`：同意采用低保真HUD先打通链路，输入锁+固定结算顺序+seed日志是必要前提。  
- 与 `prototype-qa`（预期）：同意用固定seed脚本验证“建造3类→分配→事件→结算→失败/存活”。

**冲突点（需提前约束）**
- 若数值字段未定（尤其预测值与失败计数），UI无法稳定做“即时+预测”反馈。  
- 若技术侧不提供“不可操作原因”字段（资源不足/人口不足/回合锁定），按钮态会不透明，影响可理解性。  
- 若过早加入重视觉动效，会压缩性能与可读性，影响20-30分钟稳定验证。

### 3) 下一步建议（必须可执行）

   为三类关键操作（建造/拆除/分配）定义统一状态：可用、禁用、资源不足、已锁定、执行成功；每种状态对应固定文案与颜色。

   以3条快速指标验收UI是否达标：  
   - 新玩家3分钟内完成首次“建造+分配+结束回合”；  
   - 关键操作点击层级不超过2层；
