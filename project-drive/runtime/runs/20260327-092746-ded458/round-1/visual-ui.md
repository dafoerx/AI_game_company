### 1) 本角色判断（visual-ui）

1. **同意以 TASK-006 为当前唯一活跃任务，并执行 MVP 范围冻结**：评审包只覆盖“必做6项”，UI不再新增引导、科技树、存档等入口与交互。  
2. **评审版UI目标应定义为“可读稳定”而非“风格满配”**：赛博风格保留基调，但默认启用低特效模板，确保关键信息（资源、告警、回合反馈）3秒内可识别。  
3. **15分钟演示必须有固定信息动线**：总览读数 → 人口分配 → 建造决策 → 事件响应 → 回合复盘，避免页面跳转过多导致评审理解断层。  
4. **UI层也需要“冻结与门禁”**：建议建立 `UI-RC1`，冻结后仅允许修复影响演示理解的缺陷（错层、误导文案、告警不显著、关键按钮不可达）。  
5. **评审材料里必须显式写“可用性风险与缓解”**：高信息密度、色彩疲劳、术语门槛是已知风险，要有对应规避策略（层级、对比、文案简化、低特效开关）。

### 2) 对其他角色的冲突点/共识点

- **与 producer**
  - 共识：范围收口、RC1冻结、以评审可通过为目标。  
  - 冲突点：若“视觉完整度”权重过低，可能造成界面不一致、评审观感割裂；建议保留**最低一致性基线**（色板、字体层级、告警样式统一）。

- **与 systems-economy**
  - 共识：需要展示“净变化/风险趋势”来支撑策略判断。  
  - 冲突点：若强推“下回合完整预测”，实现成本偏高；UI建议本轮先落地“本回合净变化 + 下回合风险提示（高/中/低）”作为保底。

- **与 web-tech-lead**
  - 共识：固定种子、低特效、可复盘日志是演示稳定核心。  
  - 冲突点：仅有技术日志不够评审阅读，需补“可视化事件摘要卡”（事件名、影响资源、处置建议、严重级别）。

- **与 prototype-qa（预期协作）**
  - 共识：冒烟+回归+已知问题分级可执行。  
  - 冲突点：若只测“能不能点通”，会遗漏“看不懂但能点”的可用性问题；需加入可读性指标（3秒识别率、3分钟上手率）。
