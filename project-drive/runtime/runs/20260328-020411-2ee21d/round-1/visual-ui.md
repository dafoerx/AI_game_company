1) **本角色判断（visual-ui）**

1. **方向基本正确，可支撑MVP，但还不能直接“无歧义落地UI”。**  
   目前规则偏“系统可算”，还差“玩家可懂”的展示映射（为什么满意度掉、为什么今天看着收钱却利润低）。

2. **我支持“主界面仅4核心指标”的收敛方案。**  
   营业中固定显示：`现金 / 满意度 / 库存健康度 / 当日利润`，其他中间变量（预算偏离、缺货因子、等待回合）放二级信息层（悬浮说明或日结明细）。

3. **“现金流 vs 经营利润”双口径必须做强文案区分，否则用户会误读。**  
   建议UI命名固定为：  
   - 现金：**账户余额**（资金链）  
   - 利润：**经营利润**（本日经营质量）  
   并在日结页给一行公式拆解，避免“我明明收了钱怎么还是亏”的认知断层。

4. **满意度公式可用，但需要“原因码→可视化反馈”标准。**  
   至少要把 `等待过久 / 锅底缺货 / 菜品缺货 / 超预算` 映射成统一标签和颜色等级，不然首轮试玩很难让玩家理解决策后果。

5. **三条经营路径不仅要可复现，也要可“看懂趋势”。**  
   每条脚本需附“UI预期表现”（例如第几回合开始排队变红、何时利润转负），便于QA和设计一起校验体验一致性。

---

2) **对其他角色的冲突点/共识点**

**共识点**
- 与 `producer`：同意不扩机制，v1聚焦最小闭环，避免信息过载。
- 与 `systems-economy`：同意离散回合、固定顺序、双口径并存；公式方向可用。
- 与 `web-tech-lead`：同意先保证可复现与日志追溯，复杂中间量不进主UI。

**冲突点 / 待收敛**
- **口径冲突风险**：若不统一“现金/利润”显示文案，玩家极易误判经营状态。  
- **状态定义风险**：`wait_turns`、离队时机未定，会直接影响排队条和告警显示逻辑。  
- **反馈密度风险**：若每回合弹过多提示，会打断操作；需限定提示优先级和并发数。  
- **可解释性缺口**：目前日志是技术可读，不是玩家可读，需补“原因码→玩家文案”。
