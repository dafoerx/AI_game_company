### 1) 本角色判断（visual-ui）

1. **我认为 TASK-002 已可进入 v1 冻结与实现阶段**，UI侧不存在新的机制阻塞项。  
2. **信息架构可收敛为三层**：  
   - L1常驻：`账户余额 / 经营利润（当日） / 满意度 / 库存健康度`  
   - L2轻提示：缺货、排队超时、满意度下滑  
   - L3日结复盘：利润拆解+原因Top3  
3. **“账户余额 vs 经营利润”文案必须固定**，并在日结页显示统一公式，避免玩家误解“收钱=盈利”。  
4. **满意度必须绑定 reason_code 可视化**（至少 WAIT_LONG / BROTH_OOS / ITEM_OOS / OVER_BUDGET），否则玩家看得见结果但看不懂原因。  
5. **三种经营结果（亏损/保本/盈利）不仅要可触发，还要可读**：每条脚本需标注UI预期拐点（何时告警、何时转亏/转盈）。
