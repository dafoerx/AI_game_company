### 1) 本角色判断（web-tech-lead）

1. **TASK-004技术上可在MVP周期内落地，但前提是先冻结“事件口径+字段语义”**。  
   当前最大风险不是代码难度，而是“同一业务事件在不同模块被不同口径处理”，会直接导致错账和QA无法复现。

2. **建议采用“轻量状态机 + 单一数据源（Store）+ 事件日志”实现闭环**。  
   状态机仅保留最小营业阶段：`采购中 -> 营业中 -> 日结中 -> 次日准备`，避免过度工程化。

3. **模块拆分建议固定为5块**：  
   - `procurement`（采购与现金扣减）  
   - `inventory`（库存校验/扣减/损耗）  
   - `customer-flow`（排队-入座-点单-就餐-离店）  
   - `settlement`（订单结算、满意度变更）  
   - `daily-report`（日结聚合与跨日快照）  
   这样可并行开发且便于定位问题。

4. **“实时更新”必须定义为事件级刷新，不是帧刷新**。  
   至少在`采购确认、上菜成功、结账完成、顾客离店、日结完成`5类事件后，现金/满意度/库存立即更新并记录日志。

5. **必须有降级方案**：  
   - UI未完成时，用调试面板展示三指标+事件流，保证可验收；  
   - 动效或复杂交互影响进度时可关闭，不影响经营逻辑；  
   - 若顾客AI流程不稳定，先用固定脚本驱动客流，确保3日可玩链路优先达成。

### 2) 对其他角色的冲突点/共识点

- **与 producer**  
  - 共识：先打通“可玩链路”，严格控范围，不引入随机事件等扩展。  
  - 冲突点：如果在基线文件未统一前直接开工，后续返工概率高；需先修复任务卡与活跃任务冲突。

- **与 systems-economy**  
  - 共识：支持“点单做可供检查、上菜才扣库存、结账才入账”的口径。  
  - 冲突点：规则文档目前仍缺少“边界条件真值表”（现金不足、库存临界、满意度上下限触发时机），这部分不补齐会导致实现分歧。

- **与 visual-ui**  
  - 共识：三指标常驻可读性优先于美术表现。  
  - 冲突点：若UI稿延后，技术侧需先上占位版HUD；不应等待完整视觉资源才联调。

- **与 prototype-qa**  
  - 共识：固定脚本对账 + 随机稳定性双轨验证是必要的。  
  - 冲突点：若无统一事件ID与日志格式，QA虽能发现问题但难以快速定位模块责任。
