### 1) 本角色判断（producer）

1. **先做“可跑通”，暂不做“高拟真”**  
   TASK-002 在 M1 属于 P0，必须优先保证：状态可计算、事件可触发、送养可判断。  
   裁决：本轮只做 **统一 0-100 数值轴 + 线性可解释公式**，不引入复杂人格成长树/隐式 AI 行为模型。

2. **验收边界明确化（防范围漂移）**  
   以验收标准为硬门槛：  
   - 3 类动物（狗/猫/兔）初始参数表（完整字段）  
   - ≥9 个心结事件（建议每类至少 3 个，允许跨物种复用模板）  
   - 可领养度规则可直接驱动“可送养/观察中/不可送养”判断，并可前端展示  
   以上缺一项即不通过。

3. **状态模型必须“可前端读取”而非只在文档里可读**  
   裁决：输出必须同时包含 **表格版 + JSON 字段契约**（字段名、范围、更新方向、触发来源）。仅有设计叙述不可验收。

4. **失败惩罚要有压力但不能劝退**  
   情绪主题重，若惩罚过重会导致玩家不敢互动。  
   裁决：失败惩罚上限需被钳制（如单次失败不允许让可领养度跌落超过一个状态档位），并提供“恢复路径”（连续2-3次正向互动可挽回）。

5. **当前文档基线存在冲突，需先统一任务源**  
   `00-active-context` 显示 TASK-002 为当前活跃；README 仍写 TASK-001。  
   裁决：本轮以 `00-active-context` 为准，立即补建/更新 TASK-002 任务卡，避免团队按错任务推进。
