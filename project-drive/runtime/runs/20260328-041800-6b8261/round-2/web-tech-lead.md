### 1) 本角色判断（web-tech-lead）

1. **TASK-003 技术上可在本迭代落地，但必须走“单局原型优先”路线**：采用单页线性流程（接收→照护→状态查看→送养确认→结果反馈），不做通用化大框架，不并行开发非闭环功能。  
2. **状态保存与刷新恢复是本任务的硬门槛**：需要实现 `SessionStore`（本地持久化）+ `Checkpoint`（关键节点存档）+ `恢复弹窗` 三件套，否则验收标准不成立。  
3. **流程时长 10-15 分钟可控**：通过“动作预算+步骤上限”实现（如照护阶段最多 3 轮互动、单次小游戏限时、送养流程强约束 CTA），避免玩家超时或卡关。  
4. **模块拆分建议最小化**：  
   - `flow-engine`（流程状态机）  
   - `state-engine`（心结/信任/应激变更）  
   - `interaction-module`（观察对话+小游戏）  
   - `adoption-module`（匹配与确认）  
   - `persistence-module`（存档/恢复/版本兼容）  
   - `telemetry`（埋点与调试日志）  
5. **主要风险与降级方案明确**：  
   - `localStorage` 不可用或损坏 → 降级为内存会话并提示“本局无法跨刷新保存”；  
   - 数值规则未冻结 → 用可配置 JSON 表驱动，先保证可解释反馈；  
   - 资源加载慢 → 占位图+延迟加载，优先保证流程可跑通。
