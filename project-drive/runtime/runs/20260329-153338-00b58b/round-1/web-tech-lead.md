### 1) 本角色判断（web-tech-lead）

1. **技术上可在 M2 前半落地**，前提是严格执行“配置驱动 + 确定性计算 + 最小事件类型”，不引入随机评分和复杂状态机扩展。  
2. **当前最大工程风险不是算法，而是数据契约未冻结**：家庭画像、动物特征、发布标签、评分拆解、回访触发条件必须先统一字段与枚举，否则前后端会反复改。  
3. **建议按三模块拆分实现**：  
   - 匹配引擎（硬门槛+加权分）  
   - 可解释层（Top3影响因子+禁忌原因+建议）  
   - 回访触发器（T+7/T+21，正向/中性）  
4. **禁忌必须“硬拦截优先”**，不能被高分覆盖；UI 要显示“不可推荐原因”，避免误导玩家。  
5. **需要明确降级方案**：配置加载失败时回退到内置默认规则与中性回访，保证演示闭环不断。

### 2) 对其他角色的冲突点/共识点

- **与 producer**
  - 共识：先闭环、后扩展；不做ML、不做长期家庭关系模拟。  
  - 冲突点：D3-D5 同时完成引擎+UI+事件联调偏紧，建议拆成“D3引擎跑通、D4界面接入、D5修复回归”。

- **与 systems-economy**
  - 共识：7因子以内、标签只做权重修正、禁忌硬约束、T+7/T+21双节点。  
  - 冲突点：还缺少**精确字段和边界定义**（如69/70、84/85阈值、同分排序规则），这会直接影响实现与QA复现。

- **与 visual-ui**
  - 共识：展示“分数+关键因子+禁忌警示”即可满足可解释性。  
  - 冲突点：若额外加入重情绪演出（长动效/多媒体）会挤占主流程开发，建议本轮只做轻量卡片。

- **与 prototype-qa**
  - 共识：核心用例覆盖禁忌拦截、同分排序、边界分、双节点触发。  
  - 冲突点：若逻辑包含随机扰动会导致不可复现；建议MVP阶段核心逻辑全确定性。
